# Semantic Sync — 구조적 변경에서 사용자 의도를 읽는 파이프라인

**Date**: 2026-03-13
**Status**: 설계 (구현 전)
**선행**: D-015 (Diff Pipeline Hybrid), D-016 (Sub-agent 시각화 전문가)

---

## 1. 문제 정의

### 시나리오

Claude Code가 mindmap을 생성: "프로젝트" 아래에 [기획, 개발, 리서치]를 **같은 위계**로 배치.
사용자가 브라우저에서 "리서치"를 "기획" 아래로 드래그.

**현재 diff 출력:**
```
변경사항:
  Moved: 리서치 — 프로젝트 → 기획
```

**사용자의 실제 의도:**
"리서치는 독립 활동이 아니라 기획의 하위 활동이다"

### 핵심 질문

> 현재 파이프라인이 구조적 변경(Moved)에서 의미적 의도(종속 관계 판단)를 **안정적으로** 추출할 수 있는가?

### 진단

| 역량 | 현재 상태 | 평가 |
|------|----------|------|
| 구조적 변경 감지 | diff.ts — added/removed/modified/moved | **OK** |
| 시각화 유형별 의미 구분 | diff에 유형 정보 없음 | **Gap** |
| 대화 맥락 기반 의도 추론 | translator가 할 수 있으나 비결정적 | **Weak** |
| 추론 결과 반영 | 한 줄 요약으로 nuance 손실 가능 | **Weak** |

**결론**: 이론적으로 가능하지만, **안정적이지 않다**. 같은 "moved" 변경에 대해 세션마다 다른 해석이 나올 수 있음.

## 2. 의미적 해석 레벨

사용자의 구조 변경이 Claude에게 전달되는 과정에서 4개 레벨의 해석이 존재한다:

### Level 0 — Raw Structural Diff (현재 diff.ts)

```
Moved: 리서치 — 프로젝트 → 기획
```

순수 구조 변경 사실. 의미 없음. ID와 parentId 비교 결과일 뿐.

### Level 1 — Type-Aware Semantic Annotation

```
위계 변경: 리서치가 기획의 하위 개념으로 재분류됨 (mindmap parent 변경)
```

시각화 유형(mindmap/flowchart)에 따라 같은 "moved"가 다른 의미를 가짐:
- **Mindmap**: parent 변경 = **위계/종속 관계** 변경
- **Flowchart**: parent 변경 = **흐름/순서** 변경
- **Free geometry**: position 변경 = **시각적 그룹핑** (의미 약함)

이 레벨은 **규칙 기반**으로 안정적 보장 가능.

### Level 2 — Contextual Intent Inference (현재 vync-translator 영역)

```
사용자가 리서치를 기획의 하위로 이동 → 리서치를 기획 단계의 일부 활동으로 판단한 것으로 보임.
프로젝트 초기 논의에서 3개 독립 축이었으나, 사용자는 리서치를 기획에 종속시킴.
```

대화 맥락 + Level 1 annotation을 결합한 LLM 추론. **비결정적**이지만 가장 가치 있는 해석.

### Level 3 — Confirmed Intent (사용자 확인)

```
사용자: "맞아, 리서치는 기획 팀이 수행하는 활동이야"
```

가장 정확하지만, 매번 물어보면 대화 흐름이 끊김 (Vync 철학 위배).

### 현재 파이프라인 레벨 매핑

```
diff.ts          → Level 0 (구조적 사실만)
vync-translator  → Level 0 → Level 2로 점프 (Level 1 없이)
메인 세션        → Level 2 요약 수신
```

**Gap**: Level 0 → Level 2 사이에 Level 1이 없어서, translator가 시각화 유형별 의미를 직접 추론해야 함. 이 추론의 안정성이 보장되지 않음.

## 3. 시각화 유형별 해석 규칙

구조적 변경의 의미는 시각화 유형에 따라 달라진다. 이 규칙은 diff annotation 또는 translator prompt에 포함되어야 한다.

### Mindmap

마인드맵에서 구조 = 개념의 위계. 모든 구조 변경이 **개념적 관계**의 변경을 의미한다.

| 변경 유형 | 구조적 사실 | 의미적 해석 |
|----------|-----------|-----------|
| Moved: B — root → A | B의 parent가 root에서 A로 변경 | **B는 A에 종속되는 개념** (독립 → 하위) |
| Moved: B — A → root | B의 parent가 A에서 root로 변경 | **B는 A에서 독립된 개념** (하위 → 독립) |
| Moved: B — A → C | B의 parent가 A에서 C로 변경 | **B는 A가 아니라 C의 하위 개념** (재분류) |
| Added: B (under A) | B가 A 아래에 신규 생성 | **A의 새로운 하위 개념/요소 추가** |
| Removed: B (was under A) | B가 삭제됨 | **A에서 B 개념 제거 (불필요 판단)** |
| Modified: "X" → "Y" | 텍스트 변경 | **개념의 재정의/명확화** |

**복합 변경 패턴**:
- Moved + Modified: 재분류 + 재정의 → **개념 자체를 재해석**
- 여러 노드가 동시에 같은 parent로 Moved: **그룹화/범주화 의도**
- 하위 노드 전체 Removed + 상위 노드 Modified: **추상화/단순화 의도**

### Flowchart (Geometry + Arrow-line)

플로우차트에서 구조 = 프로세스의 흐름. 변경은 **흐름/순서/의존성**의 변경.

| 변경 유형 | 의미적 해석 |
|----------|-----------|
| Added: 새 노드 + 화살표 | **새 단계/분기 추가** |
| Removed: 노드 + 연결된 화살표 | **단계 제거 (불필요 판단)** |
| Modified: 텍스트 변경 | **단계의 내용 재정의** |
| Arrow 추가/제거 | **의존성/흐름 관계 변경** |

### Free Geometry (캔버스 위 자유 배치)

구조적 관계가 약함. 변경의 의미 해석에 주의 필요.

| 변경 유형 | 의미적 해석 |
|----------|-----------|
| 위치 변경 (layout only) | **시각적 정리일 가능성 높음** (diff가 무시) |
| 그룹에 포함/제외 | **범주화/탈범주화** |
| 텍스트 변경 | **내용 재정의** |

## 4. 파이프라인 개선 방향

### 아키텍처 결정: Sub-agent 중심 유지

의미 해석의 주체는 **vync-translator sub-agent**로 유지한다 (D-013 연장).

**근거:**
- 메인 세션의 context window 보호 (D-013의 핵심 동기)
- translator가 이미 "시각화 전문가"로서 맥락+diff 결합 능력 보유 (D-016, PoC 3/3 PASS)
- 해석 품질 향상은 **입력 품질 개선**으로 달성 (아키텍처 변경 불필요)

### 개선 경로: 입력 품질 강화

현재 파이프라인의 약점은 translator에게 전달되는 정보의 질이다:

```
[현재]
diff: "Moved: 리서치 — 프로젝트 → 기획"  ← Level 0, 유형 정보 없음
맥락: "프로젝트 계획 논의 중" (2-5문장)   ← 얕을 수 있음

[개선 후]
diff: "위계 변경: 리서치 → 기획 하위 (mindmap에서 parent 변경 = 종속 관계)"  ← Level 1
맥락: "프로젝트 계획 논의 중" (2-5문장) + 시각화 유형별 해석 가이드       ← 가이드 포함
```

### 4-A: diff.ts에 Semantic Annotation 추가

**현재 DiffChange:**
```typescript
interface DiffChange {
  kind: 'added' | 'removed' | 'modified' | 'moved';
  id: string;
  text: string;
  detail: string;  // "Moved: 리서치 — 프로젝트 → 기획"
}
```

**개선안:**
```typescript
interface DiffChange {
  kind: 'added' | 'removed' | 'modified' | 'moved';
  id: string;
  text: string;
  detail: string;
  semanticHint?: string;  // 시각화 유형별 의미 annotation
}
```

**semanticHint 생성 규칙** (mindmap의 경우):
- `moved` + to child: `"위계 변경: {text}가 {toParent}의 하위 개념으로 재분류됨"`
- `moved` + to root: `"독립화: {text}가 {fromParent}에서 분리되어 독립 개념으로"`
- `moved` + sibling transfer: `"재분류: {text}가 {fromParent}가 아닌 {toParent}의 하위로"`
- `added`: `"개념 추가: {text}가 {parent}의 새 하위 요소로"`
- `removed`: `"개념 제거: {text}가 {parent}에서 삭제됨"`
- `modified`: `"재정의: {oldText} → {newText}"`

**복합 패턴 감지:**
- 여러 노드가 같은 parent로 moved → `"그룹화: [{texts}]가 {parent} 하위로 통합"`
- 하위 트리 전체 removed → `"단순화: {parent}의 하위 구조 제거"`

### 4-B: Translator Prompt에 해석 가이드 추가

vync-translator의 Read 절차에 §3의 해석 규칙을 참조하도록 명시:

```markdown
### Read
1. 전달받은 diff 결과를 분석
2. **semanticHint가 있으면 이를 기반으로**, 없으면 구조+유형에서 직접 추론
3. 대화 맥락과 결합하여 "사용자의 생각이 어떻게 변했는지" 번역
4. 변화의 **방향성** 포착:
   - 구조 변경이 시사하는 사고의 전환 (예: "독립 개념 → 종속 개념")
   - 복합 변경에서 드러나는 의도 패턴 (예: "실행 중심으로 구조 이동")
5. 반환: 의미적 번역 (변화 요약 + 추론된 방향성)
```

### 4-C: Read 반환 포맷 확장 (구조화된 반환)

**적용 범위**: Read만 확장. create/update는 현행 한 줄 요약 유지 (Claude가 실행한 것이므로 의도 추론 불필요).

**현재 Read 반환:**
```
"변화 요약: 개발 프로세스에 테스팅이 추가되고, 기획의 리서치가 축소됨."
```

**개선: 4-필드 구조화 반환 (~3-5줄)**
```
요약: 리서치가 기획 하위로 이동, 테스팅이 개발에 추가됨
의도: 리서치를 독립 활동이 아닌 기획의 일부로 판단. 실행 중심으로 구조 이동 중.
확신: 높음 (mindmap 위계 변경 = 종속 판단)
제안: 개발 쪽도 비슷하게 세분화?
```

**필드 정의:**
| 필드 | 설명 | 용도 |
|------|------|------|
| 요약 | 구조적 변경의 사실적 기술 | 메인 세션이 사용자에게 전달 |
| 의도 | 변경이 시사하는 사용자의 생각/판단 추론 | 메인 세션의 대화 맥락 이해 |
| 확신 | 높음/중간/낮음 + 근거 | 메인 세션의 모호성 처리 전략 결정 |
| 제안 | 추론된 의도 기반 다음 행동 제안 (optional) | 메인 세션이 자연스러운 확인에 활용 |

**확신 레벨별 메인 세션 행동:**
- **높음**: 추론을 대화에 자연스럽게 반영 + 제안 활용
- **중간**: 추론을 언급하되 확인 포함 ("~것 같은데, 이 방향이 맞으면 ~할까요?")
- **낮음**: 구조적 사실(요약)만 전달, 의도 추론 언급하지 않음

**변경 없음일 때:**
```
요약: 변경 없음
```
(의도/확신/제안 생략)

**메인 세션 활용 예시:**

확신 높음:
```
translator: "요약: 리서치가 기획 하위로 이동\n의도: 기획의 일부로 판단\n확신: 높음\n제안: 개발 쪽도 세분화?"
→ 메인: "기획의 세부 활동으로 리서치를 보시는 것 같은데, 개발 쪽도 정리할까요?"
```

확신 중간:
```
translator: "요약: A→B 하위, C→D 하위로 이동\n의도: 쌍별 그룹화\n확신: 중간 (동시 다중 이동, 정리 가능성)\n제안: 이 구분으로 진행?"
→ 메인: "A-B, C-D로 묶으셨네요. 이 구분으로 진행할까요?"
```

확신 낮음:
```
translator: "요약: 노드 3개 위치 약간 조정\n의도: 시각적 정리로 추정\n확신: 낮음\n제안: 없음"
→ 메인: "구조에 약간 변화가 있네요." (간략히 언급, 의도 추론 없이)
```

## 5. 모호성 처리 전략

### 원칙: 추론 + 자연스러운 확인

Vync의 핵심 철학은 **대화 흐름을 끊지 않는 것**. 의도가 모호할 때도 이를 지킨다.

### 모호성 유형별 처리

**Type A: 명확한 의도** (높은 확신)
- 예: mindmap에서 "리서치"를 "기획" 아래로 이동
- 처리: 추론 결과를 대화에 자연스럽게 반영
- 확인 불필요

**Type B: 추론 가능하지만 확신 부족** (중간 확신)
- 예: 여러 노드를 동시에 재배치 (정리인지 재분류인지)
- 처리: **추론 + 대화 흐름 안에서 확인**
- 패턴: "[추론 내용]. 이 방향이 맞으면 [다음 행동 제안]할까요?"
- 예시: "기획 쪽에 리서치와 인터뷰를 묶으셨네요. 기획의 세부 활동으로 정리하시는 것 같은데, 개발 쪽도 비슷하게 세분화할까요?"

**Type C: 의미 판단 어려움** (낮은 확신)
- 예: 레이아웃만 약간 조정 (diff가 감지 못하거나 layout 필드만 변경)
- 처리: 구조적 사실(요약)만 **간략히 언급**, 의도 추론은 하지 않음
- 예: "구조에 약간 변화가 있네요." — 변경 사실은 전달하되 해석 없이
- **주의**: 완전히 무시하지 않음. 사용자의 의도적 변경을 놓치는 것보다 간략한 언급이 안전

### 확인의 형태

**하지 말 것:**
```
리서치를 기획 아래로 옮기셨는데, 어떤 의도인가요?
1) 기획의 하위 활동으로 재분류
2) 시각적 정리
3) 다른 의도
```
→ 대화를 멈추고, 사용자에게 "시험"을 보는 느낌

**해야 할 것:**
```
리서치를 기획 하위로 옮기셨네요. 기획의 세부 활동으로 보시는 것 같은데,
그 관점에서 개발 쪽도 정리할까요?
```
→ 추론을 대화에 녹이고, 다음 행동 제안으로 자연스럽게 확인
→ 사용자가 다른 의도였으면: "아니, 그냥 자리가 좁아서 옮긴 거야" → Claude 정정

## 6. 제약 조건과 한계

### 구조적 한계

1. **Layout-only 변경 감지 불가**: diff.ts가 points/width/height를 무시하므로, 노드 크기 조정이나 위치만 바꾼 경우 감지 안 됨. 이는 의도적 설계 (noise 제거)이나, "시각적 그룹핑" 의도를 놓칠 수 있음.

2. **단일 스냅샷 한계**: .lastread가 하나뿐이므로, "A→B→C" 3단계 변경에서 중간 단계 B를 놓침. 사용자가 여러 번 수정한 후 read하면 최종 결과만 보임.

3. ~~**시각화 유형 정보 부재**~~: **D-018에 의해 해소**. "한 파일 = 한 시각화 유형" 원칙으로 혼합 캔버스를 구조적으로 방지. 파일 레벨 유형 감지(root 노드 type 확인)로 충분.

### LLM 추론 한계

4. **비결정성**: 동일한 diff + 맥락에 대해 translator가 다른 해석을 내릴 수 있음. 이는 LLM 특성이며, Level 1 annotation으로 **완화**하되 **제거**할 수 없음.

5. **맥락 전달의 한계**: 메인 세션이 "2-5문장"으로 요약한 맥락이 충분하지 않을 수 있음. 특히 긴 대화에서 사용자의 이전 발언이 누락될 수 있음.

6. **의도의 다의성**: 같은 구조 변경이 여러 의도를 가질 수 있으며, 정답이 없는 경우도 있음. "추론 + 자연스러운 확인" 전략으로 대응하되, 100% 정확도는 목표가 아님.

### 수용 가능한 한계

7. **오추론의 비용이 낮음**: Vync에서 오추론의 결과는 "대화에서 잘못된 가정" 수준. 코드가 깨지거나 데이터가 손실되지 않음. 사용자가 한마디로 정정 가능. 따라서 **정확도보다 자연스러움이 우선**.

## 9. 설계 결정

### D-017: Semantic Sync (의미적 동기화)

- **결정**: diff에 시각화 유형별 semantic annotation 추가 + translator가 맥락과 결합하여 사용자 의도 추론 + 모호시 "추론 + 자연스러운 확인"
- **해석 주체**: Sub-agent (vync-translator) — D-013 연장
- **모호성 처리**: 추론 결과를 대화 흐름에 녹여서 자연스럽게 확인. 명시적 질문 지양.
- **대안 1**: 메인 세션 직접 해석 — context window 비용 (기각)
- **대안 2**: Hybrid (diff annotation + 메인 해석) — sub-agent 역할 축소 (기각)
- **대안 3**: 항상 명시적 확인 — 대화 흐름 중단 (기각)
- **근거**: 오추론 비용이 낮음 (자연어 정정 한마디). 자연스러움 > 정확도.
- **재검토 조건**: semantic hint가 translator 추론 품질을 유의미하게 향상시키지 못할 경우
- **상태**: 설계 완료, 구현 대기
