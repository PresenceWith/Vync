## 1. 문제 정의

### 1.1 현상

Semantic Sync 회귀 테스트(3라운드)에서 translator의 확신 레벨이 **전 라운드 과대평가**됨.

| 라운드 | 유저 발화 | 기대 확신 | 실제 확신 |
|--------|----------|:--------:|:--------:|
| A | "하나 더 손봤어" | 중간~높음 | **높음** |
| B | "도구 쪽도 좀 정리했어" | 중간 | **높음** |
| C | "구조를 좀 다르게 봐야 할 것 같아서... 바꿔봤어" | 낮음~중간 | **높음** |

전체 채점: 8/15 (53%) → FAIL. F3(확신) 0/3, F4(의도) 1/3, F5(응답) 1/3.

### 1.2 근본 원인 — 책임 할당의 구조적 비대칭

확신 판단에는 2축이 필요하다:

```
확신 = 행위 명확성 × 동기 명확성
```

- **행위 명확성**: "무엇이 변경되었는가"의 구조적 명확성 → diff + 시각 구조 이해 필요
- **동기 명확성**: "왜 변경했는가"의 대화적 명확성 → 유저 원문 + 전체 대화 맥락 필요

현재 파이프라인은 **두 축 모두 translator에 할당**하고 있으나:

| 축 | 필요 정보 | Translator 보유 | 메인 세션 보유 |
|----|----------|:--------------:|:------------:|
| 행위 명확성 | diff + 시각 구조 | **O** | X (이해 불가) |
| 동기 명확성 | 유저 원문 + 대화 맥락 | X (요약본만) | **O** |

**Translator는 동기 명확성을 판단할 수 없다.** 대화 전체 맥락이 없고, 매 호출마다 context window가 비워지기 때문이다. 결과적으로 translator는 행위 명확성만으로 확신을 판단하여 단일축 과대평가가 발생한다.

### 1.3 부차 원인 — 맥락 오염

메인 세션이 translator에게 전달하는 대화 맥락에 도메인 지식("Express+Vite middleware 구조")을 주입하여, translator가 유저의 모호성을 도메인 지식으로 보완 → 확신 추가 증폭. 맥락 분리(섹션 격리)로는 해결 불가 — LLM은 구획화된 추론을 하지 않는다.

## 2. 설계 — 확신 판단 책임 재할당 (B')

### 2.1 핵심 원칙

**각 축의 판단을 정보가 가장 풍부한 주체에 할당한다.**

```
Before: Translator(해석+확신) → 메인(수용)
After:  Translator(해석+해석확신) → 메인(동기평가+결합+응답)
```

### 2.2 Diff 소유권 이전

**변경**: 메인 세션이 `vync diff`를 실행하지 않는다. Translator가 직접 실행한다.

```
Before:
  메인: vync diff 실행 → diff 텍스트 획득 → translator에 전달
  Translator: 전달받은 diff 해석

After:
  메인: translator 호출 (대화 맥락만 전달)
  Translator: vync diff 직접 실행 → diff 획득 → 해석 → 결과 반환
```

**근거**: 메인 세션은 diff를 읽어도 시각적 구조를 이해할 수 없다. Translator(시각화 전문가)가 diff의 유일한 소비자가 되는 것이 깨끗한 책임 분리이다(단일 소비자 원칙).

**부수 효과**: I-001 (sub-agent `.lastread` Write 실패) 구조적 해소. Translator가 `vync diff`를 Bash로 실행하면 `.lastread` 갱신이 원자적으로 처리되어, 직접 Write할 필요 없음.

### 2.3 Translator 변경 (`agents/vync-translator.md`)

#### 반환 포맷 — `확신` → `해석 확신`, `의도` → `해석`

```
# Before
요약: <구조적 변경의 사실적 기술>
의도: <변경이 시사하는 사용자의 생각/판단 추론>
확신: <높음|중간|낮음> (<근거>)
제안: <추론 기반 다음 행동 제안, optional>

# After
요약: <구조적 변경의 사실적 기술>
해석: <이 diff의 구조적 해석 — "이 변경은 X를 시사함" 수준>
해석 확신: <높음|중간|낮음>
제안: <추론 기반 다음 행동 제안, optional>
```

- `의도` → `해석`: "사용자의 의도 추론"이 아닌 "diff의 구조적 해석"으로 영역 명확화. 메인 세션이 이 `해석`과 대화 맥락을 결합하여 최종 `의도`를 유저에게 전달한다.
- `해석 확신`: 유저 동기 판단 없이, 구조적 해석의 명확성만 평가.

#### 확신 기준 교체 — 가능한 해석의 수로 판단

```
# Before (단일축 — 변경 복잡성)
- 높음: mindmap 위계 변경, 명확한 추가/삭제
- 중간: 다중 동시 이동, 복합 변경
- 낮음: 의미 불분명한 소규모 변경

# After (구조적 해석 명확성 = 가능한 해석의 수)
- 높음: 가능한 구조적 해석이 1개뿐
  (예: "A가 B 하위에서 C 하위로 이동" = 재분류. 해석 하나.)
  (예: "D 노드 추가" = 개념 추가. 해석 하나.)
- 중간: 가능한 구조적 해석이 2-3개
  (예: "3개 노드가 동시에 다른 부모로 이동" = 재분류? 그룹화? 구조 정리?)
  (예: "카테고리명 변경 + 자식 이동" = 범위 재정의? 이름만 수정?)
- 낮음: 구조적 의미 자체가 불분명
  (예: "텍스트 '리서치' → '리서치 분석' 수정" = 구조적 의미 없음)
  (예: 레이아웃만 변경, 텍스트 소폭 수정)
```

**핵심**: "가능한 해석의 수"라는 셀 수 있는 기준을 사용하여, "동기를 판단하지 않음"이라는 부정 지시문을 제거.

#### Read 절차 변경

```
# Before
1. 전달받은 diff 결과를 분석 (직접 JSON 비교하지 않음!)
2. semanticHint가 있으면 이를 기반으로...
...

# After
1. `vync diff <file>` 실행 (Bash) → diff 결과 획득 + 스냅샷 갱신
   1a. "변경사항: 없음"이면: "요약: 변경 없음" 반환 (이후 단계 생략)
   1b. diff 실행 실패 시: "error: <설명>" 반환 (이후 단계 생략)
2. diff 결과 분석 (semanticHint 활용)
3. 대화 맥락과 결합하여 구조적 해석 도출
4. 해석 확신 판단 (가능한 구조적 해석의 수로 평가):
   - 1개 = 높음
   - 2-3개 = 중간
   - 불분명 = 낮음
5. 변화의 방향성 포착 (기존과 동일)
6. 반환: 4-필드 구조화 (요약/해석/해석 확신/제안)
7. 스냅샷은 step 1의 vync diff가 이미 처리
```

### 2.4 메인 세션 변경 (`commands/vync.md`)

#### Read 절차 — diff 실행 제거 + 최종 확신 결정 추가

```
# Before
1. 파일경로 해결
2. vync diff <file> 실행 (Bash) → 프로그래밍적 diff 결과 획득
3. 대화 맥락 요약 (2-5문장)
4. Agent tool 호출 (diff 결과 전달)
5. Sub-agent의 구조화된 반환을 활용:
   - 확신 높음 → 의도 반영 + 제안
   - 확신 중간 → 추론 + 확인
   - 확신 낮음 → 사실만 간략히

# After
1. 파일경로 해결 + 트리거 분류
2. 대화 맥락 요약 (2-5문장, 유저 원문 발화 포함)
   - 유저 발화와 논의 주제를 요약. 도메인 지식이나 기술적 해석은 포함하지 않음
3. Agent tool 호출 (diff 전달 안 함 — translator가 직접 실행):
   prompt: "## 작업: Read\n파일: <path>\n\n## 대화 맥락\n<유저 원문 포함 맥락 요약>"
4. 에러/변경 없음 처리:
   - "error:" prefix → 유저에게 알림 + 경로 재확인. 2축 매트릭스 skip.
   - "요약: 변경 없음" → 유저에게 전달. 2축 매트릭스 skip.
5. 최종 확신 결정 (메인 세션 책임):

   Step A — 트리거 분류 확인:
   - "뭐가 바뀌었어?" / 구체적 질문 → 명시적 조회.
     Step B/C 생략, 확신 무관하게 translator 해석을 사실 중심으로 완전 기술.
   - "바꿔봤어" / "정리했어" / 완료 신호 → 패시브 신호. Step B/C 진행.

   Step B — 동기 명확성 평가 (유저 발화 + 대화 맥락):
   - 높음: 유저가 변경 이유를 명시적으로 설명
     (단, 유저 설명이 diff 전체를 커버하지 않으면, 미커버 부분은 중간으로 처리)
   - 중간: 방향이나 결과만 언급, 구체적 이유 없음
   - 낮음: 맥락 없음, 또는 도메인 통념에 반하는 변경

   Step C — 최종 확신 결합:
   - 둘 다 높음 → 최종 높음
   - 둘 중 하나라도 낮음 → 최종 낮음
   - 그 외 → 최종 중간

   Step D — 파괴적 변경 가드 (Step C 후 실행):
   - translator 요약에 대량 삭제(3+ 노드 removed)가 포함되고 동기 명확성 ≤ 중간:
     → 최종 확신을 낮음으로 클램프
     → 응답에 "의도한 변경인지" 확인 포함

   Step E — 최종 확신으로 응답:
   - 높음: 해석을 대화에 자연스럽게 반영 + 제안 전달
   - 중간: 추론 언급 + 확인 포함
   - 낮음: 요약(사실)만 간략히 언급, 의도 추론은 하지 않음
```

## 3. 회귀 테스트 라운드별 예상 결과

### Round A (유저: "하나 더 손봤어")
- Translator 해석 확신: 높음 (추가+수정, 해석 1개)
- 메인 세션 동기 명확성: 중간 ("하나 더 손봤어" — 방향 암시, 이유 미제시)
- Step C: 그 외 → **최종 중간**
- 기대값 중간~높음과 **근접** ✓

### Round B (유저: "도구 쪽도 좀 정리했어")
- Translator 해석 확신: 중간 (3개 동시 변경, 해석 2-3개)
- 메인 세션 동기 명확성: 중간 ("정리했어" — 방향만, 이유 없음)
- Step C: 그 외 → **최종 중간**
- 기대값 중간과 **일치** ✓

### Round C (유저: "구조를 좀 다르게 봐야 할 것 같아서... 바꿔봤어")
- Translator 해석 확신: 높음 (이동 방향 명확, 해석 1개)
- 메인 세션 동기 명확성: 낮음 (이유 미제시, 도메인 통념에 반하는 변경)
- Step C: 하나라도 낮음 → **최종 낮음**
- 기대값 낮음~중간과 **근접** ✓ (기대보다 약간 보수적이나 fail-safe 방향)

## 4. 변경하지 않는 것

| 항목 | 이유 |
|------|------|
| diff 엔진 (`diff.ts`) | 코드 변경 없음. 시각적 변동 감지는 I-006으로 별도 추적 |
| Create/Update 절차 | read만 해당. create/update는 메인 세션이 지시를 내리는 흐름이므로 확신 판단 불필요 |
| Update flow의 diff 실행 | 메인 세션에 유지. Update의 diff는 유저 수정 확인용(상태 체크)이며, 해석/확신 판단과 무관 |
| D-013 (context window 보호) | 유지. 메인 세션은 여전히 .vync JSON을 직접 읽지 않음 |

## 7. 위험 및 한계

| 위험 | 심각도 | 완화 |
|------|:------:|------|
| Translator가 `해석 확신`도 과대평가 | 중 | "가능한 해석의 수"로 셀 수 있는 기준 제공. 과대평가 시 메인 세션의 결합 규칙에서 보정 |
| 메인 세션의 동기 명확성 평가 일관성 | 중 | 3단계(높/중/낮) 기준 + 유저 원문 대조. Translator보다 정보량 우위 |
| Translator diff 실패 후 스냅샷 비정합 | 낮-중 | diff 실행 실패 시 스냅샷 미갱신. 성공 후 translator 크래시 시 스냅샷은 전진했으나 해석 미전달 → 다음 read에서 "변경 없음" 보고. **복구**: `vync diff --no-snapshot`으로 현재 상태 확인 가능 |
| Translator의 `해석` 필드가 사실적으로 틀릴 경우 | 중 | 메인 세션은 .vync를 읽지 않으므로(D-013) 검증 불가 — **구조적 한계로 수용**. 메인 세션은 해석의 질이 아닌 확신 레벨의 적절성만 평가 |
| 동일 변경에 대한 해석 확신 비결정성 (multi-round 진동) | 낮 | 단일축(구조적 명확성)으로 범위를 좁혀 진동 폭 최소화. 메인 세션의 동기 축이 안정 앵커 역할 |
| Sub-agent cold start 시 절차 이탈 | 낮 | 번호 매긴 절차 + 구체 예시로 일관 동작 유도. Create/Update에서 이미 Bash 사용 패턴 확립 |
